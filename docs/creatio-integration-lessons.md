# Creatio Component Integration — Lessons Learned

A detailed log of what worked, what didn't, and why — for building a reference guide on developing Angular web components for the Creatio platform.

## Architecture Overview

The component (`<forecast-hierarchy>`) is an Angular 21 web component built with `@angular/elements`. There are **two integration layers** with Creatio:

1. **AMD layer** — the core Angular 21 component, wrapped in an IIFE, loaded via RequireJS. This registers the `<forecast-hierarchy>` custom element for use on any page via DOM manipulation.

2. **Module Federation layer** — a thin Angular 17 wrapper project (generated by `clio new-ui-project`), using `@creatio-devkit/common` decorators (`@CrtViewElement`, `@CrtInterfaceDesignerItem`). This makes the component visible in the Freedom UI Designer and usable declaratively in `viewConfigDiff`.

Both layers are loaded via the same `bootstrap.js` file.

**Three build targets:**
- `npm start` → standalone sandbox at http://localhost:4200 (for development)
- `npm run build:creatio` → AMD-wrapped JS file for runtime use
- `npm run build` (in `projects/forecast_hierarchy/`) → Module Federation remote module for designer integration

## What Worked

### 1. Angular Elements as Custom Elements
Standard `createCustomElement()` from `@angular/elements` works perfectly in Creatio. The custom element is registered via `customElements.define('forecast-hierarchy', el)` and can be created/used anywhere in the Creatio page via `document.createElement('forecast-hierarchy')`.

### 2. ViewEncapsulation.ShadowDom
Required by Creatio for style isolation. Works correctly. The Montserrat font `@import` must be placed **inside** the component's styles block (not in global styles) because ShadowDom blocks external stylesheets.

### 3. AMD Bootstrap Pattern
The working pattern for loading a web component into Creatio's RequireJS environment:

**bootstrap.js** — registers an AMD path:
```js
(function () {
    require.config({
        paths: {
            "forecast-hierarchy": Terrasoft.getFileContentUrl(
                "UsrComponentPackage",    // <-- must match the Creatio package name exactly
                "src/js/forecast-hierarchy.js"
            ),
        },
    });
})();
```

**Files/descriptor.json** — tells Creatio to execute the bootstrap:
```json
{
    "bootstraps": ["src/js/bootstrap.js"]
}
```

**Page schema** — declares the dependency:
```js
define("UsrTestApp_FormPage", ["forecast-hierarchy"], function() { ... });
```

### 4. IIFE Wrapper (Not AMD define)
The component JS must be wrapped in an **IIFE**, not an AMD `define()`:
```js
(function(){
  // ... Angular app code ...
})();
```

### 5. createApplication() (Not platformBrowser().bootstrapModule())
The Creatio entry point must use `createApplication()` which doesn't require a root DOM element:
```ts
import { createApplication } from '@angular/platform-browser';
import { createCustomElement } from '@angular/elements';

createApplication().then((appRef) => {
    const el = createCustomElement(Component, { injector: appRef.injector });
    customElements.define('forecast-hierarchy', el);
});
```

### 6. File System Development Mode (FSM)
Enable with: `clio turn-fsm on -e creatio-local`
- Creatio reads schemas from `Pkg/*/Schemas/` directory
- Changes to Freedom UI page schemas (JS) are picked up on page refresh
- No compilation needed for front-end schema changes

### 7. Button + Handler Pattern for Testing
Simplest way to test a component loads correctly:
```js
// In viewConfigDiff:
{
    "operation": "insert",
    "name": "UsrTestButton",
    "values": {
        "type": "crt.Button",
        "caption": "Test Component",
        "color": "accent",
        "clicked": { "request": "usr.TestButtonClicked" }
    },
    "parentName": "SideAreaProfileContainer",
    "propertyName": "items",
    "index": 1
}

// In handlers:
{
    request: "usr.TestButtonClicked",
    handler: async (request, next) => {
        const el = document.createElement("forecast-hierarchy");
        window.alert("Component loaded! Tag: " + el.tagName);
        return next?.handle(request);
    }
}
```

### 8. Freedom UI Designer Integration via Module Federation

To make a component appear in the Freedom UI Designer's component library (so users can drag-and-drop it onto pages), you need a **separate Angular 17 project** that uses webpack Module Federation with `@creatio-devkit/common`.

**Why a separate project:** The main component uses Angular 21 with Vite (esbuild), but Creatio's Module Federation requires Angular 17 with webpack. The two can't coexist in one build. The solution is a thin wrapper that hosts the custom element.

**Step 1: Generate the project with clio:**
```bash
clio new-ui-project forecast_hierarchy -v usr --package UsrComponentPackage --empty true -e creatio-local --silent
```
This creates `projects/forecast_hierarchy/` with webpack.config.js, Angular 17 deps, and Module Federation already configured.

**Step 2: Create the wrapper component** (`src/app/view-elements/forecast-hierarchy/forecast-hierarchy.component.ts`):
```typescript
import { Component, ViewEncapsulation, Input, OnChanges, SimpleChanges, ElementRef, ViewChild, AfterViewInit } from '@angular/core';
import { CrtViewElement, CrtInput, CrtInterfaceDesignerItem } from '@creatio-devkit/common';

@Component({
  selector: 'usr-forecast-hierarchy',
  template: `<div #container></div>`,
  styles: [`:host { display: block; width: 100%; }`],
  encapsulation: ViewEncapsulation.ShadowDom,
})
@CrtViewElement({
  selector: 'usr-forecast-hierarchy',
  type: 'usr.ForecastHierarchy',
})
@CrtInterfaceDesignerItem({
  toolbarConfig: {
    caption: 'Forecast Hierarchy',
    name: 'usr_forecast_hierarchy',
    icon: require('!!raw-loader?{esModule:false}!./icon.svg'),
    defaultPropertyValues: {
      label: 'Forecast Hierarchy',
    },
  },
})
export class ForecastHierarchyComponent implements AfterViewInit, OnChanges {
  @ViewChild('container', { static: true }) containerRef!: ElementRef<HTMLDivElement>;
  @Input() @CrtInput() data: any[] = [];
  @Input() @CrtInput() columns: any[] = [];
  @Input() @CrtInput() hierarchy: any = { idField: 'id', parentIdField: 'parentId', nameField: 'name' };
  @Input() @CrtInput() storageKey: string = 'forecast-hierarchy';
  private fhElement: any = null;

  ngAfterViewInit(): void {
    this.fhElement = document.createElement('forecast-hierarchy');
    this.fhElement.setAttribute('storage-key', this.storageKey);
    this.updateComponent();
    this.containerRef.nativeElement.appendChild(this.fhElement);
  }
  ngOnChanges(): void { if (this.fhElement) this.updateComponent(); }
  private updateComponent(): void {
    if (this.data) this.fhElement.data = this.data;
    if (this.columns) this.fhElement.columns = this.columns;
    if (this.hierarchy) this.fhElement.hierarchy = this.hierarchy;
  }
}
```

Key requirements:
- `@CrtViewElement` — registers the component type as `'usr.ForecastHierarchy'`
- `@CrtInterfaceDesignerItem` — adds it to the designer toolbar with icon and caption
- `require('!!raw-loader?{esModule:false}!./icon.svg')` — icon must use raw-loader (`npm install raw-loader --save-dev`)
- The wrapper creates a `<forecast-hierarchy>` custom element inside its template (the AMD-loaded core component)

**Step 3: Register in app.module.ts:**
```typescript
@CrtModule({
  viewElements: [ForecastHierarchyComponent],
})
@NgModule({
  declarations: [ForecastHierarchyComponent],
  imports: [BrowserModule],
  schemas: [CUSTOM_ELEMENTS_SCHEMA],
})
export class AppModule implements DoBootstrap {
  constructor(private _injector: Injector) {}
  ngDoBootstrap(): void {
    const element = createCustomElement(ForecastHierarchyComponent, { injector: this._injector });
    customElements.define('usr-forecast-hierarchy', element);
    bootstrapCrtModule('forecast_hierarchy', AppModule, {
      resolveDependency: (token) => this._injector.get(token as ProviderToken<unknown>),
    });
  }
}
```

**Step 4: Build and deploy:**
```bash
cd projects/forecast_hierarchy && npm run build
```
Build output goes to `packages/UsrComponentPackage/Files/src/js/forecast_hierarchy/` (configured in angular.json `outputPath`). Copy to deployed Creatio and push the package.

**Step 5: Bootstrap must load the remoteEntry.js explicitly:**

Creatio does NOT auto-discover remote modules from the file system. The `bootstrap.js` must load `remoteEntry.js` via a script tag and call Module Federation's `init()` + `get()`:

```js
(function () {
    /* AMD: register the core custom element */
    require.config({
        paths: {
            "forecast-hierarchy": Terrasoft.getFileContentUrl(
                "UsrComponentPackage",
                "src/js/forecast-hierarchy.js"
            ),
        },
    });

    /* Module Federation: load the designer wrapper */
    var remoteEntryUrl = Terrasoft.getFileContentUrl(
        "UsrComponentPackage",
        "src/js/forecast_hierarchy/remoteEntry.js"
    );
    var script = document.createElement("script");
    script.src = remoteEntryUrl;
    script.type = "text/javascript";
    script.onload = function () {
        if (window.forecast_hierarchy) {
            window.forecast_hierarchy.init(__webpack_share_scopes__.default || {});
            window.forecast_hierarchy.get("./RemoteEntry").then(function (factory) {
                factory();
            }).catch(function(e) {
                console.warn("Remote module init:", e);
            });
        }
    };
    document.head.appendChild(script);
})();
```

**This is the critical discovery:** without this explicit script-tag loading in bootstrap.js, the remoteEntry.js is never fetched and the component never appears in the designer. Creatio only auto-loads bootstraps from `_FileContentBootstraps.js`, it does NOT scan for `remoteEntry.js` files.

## What Did NOT Work

### 1. Named AMD define() — "Mismatched anonymous define()"
**Problem:** Wrapping the JS as `define("forecast-hierarchy", [], function() { ... })` caused RequireJS error:
```
Mismatched anonymous define() module: function(){return tr}
```
**Why:** RequireJS loads the file via a `<script>` tag. When the script contains a named `define()`, RequireJS can't match it to the requested module name because the URL-based name doesn't match. Named defines only work when the module is inlined/concatenated.
**Fix:** Use an IIFE wrapper instead. The bootstrap's `require.config({ paths: ... })` tells RequireJS where to find the file, and the IIFE just executes and registers the custom element.

### 2. @creatio-devkit/common SDK — Lodash Conflict
**Problem:** Including `@creatio-devkit/common` decorators (`@CrtViewElement`, `@CrtInterfaceDesignerItem`) in the component caused the SDK's bundled lodash to conflict with Creatio's global lodash:
```
TypeError: (0, In.isEmpty) is not a function
```
**Why:** The `@creatio-devkit/common` package bundles lodash internally. When our build includes it, two lodash versions compete, and Creatio's own code breaks because `_.isEmpty` gets overwritten.
**Fix:** Remove `@creatio-devkit/common` imports from the component. The decorators are only needed for the Module Federation approach (remote modules), not for the AMD/custom-element approach. Keep the SDK as a dev dependency for future use.

### 3. platformBrowser().bootstrapModule() — No Root Element
**Problem:** The initial Creatio entry point used:
```ts
platformBrowser().bootstrapModule(CreatioModule)
```
This requires a root DOM element (like `<app-root>`) which doesn't exist in Creatio pages.
**Fix:** Use `createApplication()` instead, which creates a headless Angular application without needing a root element.

### 4. Symlinks in Creatio Content Directory
**Problem:** Symlinks from `conf/content/UsrComponentPackage/src/js/forecast-hierarchy.js` → source file served only 100 bytes (truncated).
**Why:** Creatio's static file server (Kestrel) doesn't follow symlinks properly — it reads the symlink metadata size instead of the target file size.
**Fix:** Use actual file copies, not symlinks. The `build-creatio.js` script copies the built file to both `creatio-package/` and the deployed `conf/content/` directory.

### 5. Package Name Mismatch in bootstrap.js
**Problem:** `bootstrap.js` referenced `"ForecastHierarchy"` as the package name in `Terrasoft.getFileContentUrl()`, but the actual Creatio package was named `"UsrComponentPackage"`.
**Why:** The URL pattern is `conf/content/{PackageName}/src/js/...` — wrong package name = 404.
**Fix:** The first argument to `Terrasoft.getFileContentUrl()` must exactly match the Creatio package name.

### 6. clio push-pkg to Cloud Trial
**Problem:** `clio push-pkg` failed against the Creatio cloud trial with JSON parse errors during upload.
**Why:** Cloud trials restrict the package upload API endpoint — the server returns HTML error pages instead of JSON responses.
**Fix:** Use local Creatio deployment instead, or upload packages manually through the Creatio Configuration UI.

### 7. clio install-gate on Cloud Trial
**Problem:** Same upload failure as push-pkg — cliogate couldn't be installed via clio.
**Fix:** Download `cliogate.gz` from clio's installation directory (`~/.dotnet/tools/.store/clio/*/tools/net8.0/any/cliogate/`) and upload it manually through Creatio's Configuration > Install from file.

### 8. pg_restore Version Mismatch
**Problem:** Local `pg_restore` was version 12 (from Postgres.app), but the Creatio backup was created with PostgreSQL 17.
```
pg_restore: error: unsupported version (1.15) in file header
```
**Fix:** Install latest PostgreSQL client tools via `brew install libpq` and configure `PgToolsPath` in clio's `appsettings.json` to `/opt/homebrew/opt/libpq/bin`.

### 9. PostgreSQL Container Version
**Problem:** PostgreSQL 16 container didn't support `transaction_timeout` parameter from the backup:
```
ERROR: unrecognized configuration parameter "transaction_timeout"
```
**Fix:** Use PostgreSQL 17+ container: `podman run ... postgres:17-alpine`

### 10. .NET Runtime Version
**Problem:** Creatio 8.3.3 needs .NET 8 runtime, but only .NET 10 was installed.
**Fix:** Install .NET 8 via `brew install dotnet@8`. Start Creatio with explicit runtime:
```bash
DOTNET_ROOT="/opt/homebrew/opt/dotnet@8/libexec" \
  /opt/homebrew/opt/dotnet@8/libexec/dotnet Terrasoft.WebHost.dll
```

### 11. Remote Module Not Discovered Automatically
**Problem:** After deploying the Module Federation build (remoteEntry.js + chunks) to the Creatio package and pushing it, the component did not appear in the Freedom UI Designer. The remoteEntry.js was never loaded by the browser.
**Why:** Creatio does NOT auto-discover remote modules from the file system. The `_FileContentBootstraps.js` only triggers AMD bootstraps. There is no `SysRemoteModule` table or config file that maps remote modules. The shell app only loads remotes that are explicitly registered via `bootstrapCrtModule()` which requires the remoteEntry to be loaded first.
**Fix:** The bootstrap.js must explicitly create a `<script>` tag to load remoteEntry.js, then call `window.forecast_hierarchy.init()` and `.get("./RemoteEntry")` to trigger the Module Federation handshake and execute `bootstrapCrtModule()`.

### 12. @creatio-devkit/common Only Works with Angular 17 + Webpack
**Problem:** Attempted to use `@CrtViewElement` and `@CrtInterfaceDesignerItem` decorators in the Angular 21 (Vite/esbuild) build. The decorators bundled lodash causing conflicts, and the Module Federation protocol doesn't work without webpack.
**Why:** `@creatio-devkit/common` is designed for Angular 17 with webpack Module Federation. It bundles lodash-es and uses reflect-metadata, both of which cause issues in Angular 21's esbuild pipeline. Module Federation itself is a webpack feature not available in Vite.
**Fix:** Use a separate Angular 17 project (generated by `clio new-ui-project`) that wraps the core custom element. The Angular 21 project remains for the core component logic; the Angular 17 project is only for Creatio integration metadata.

### 13. DOM Insertion of Component via Handler
**Problem:** Tried to insert the `<forecast-hierarchy>` element into the Creatio page by finding a container via `document.querySelector()` in a button handler. The selectors didn't match because Freedom UI components have generated IDs.
**Why:** Freedom UI renders components with dynamic IDs (e.g., `t-comp82`), not the schema element names. CSS class names are also minified/generated.
**Fix:** For reliable DOM insertion, either: (a) use the Freedom UI Designer to place the component via `viewConfigDiff` with a known container, or (b) use `document.querySelector("[id*='SomePartialId']")` for partial matching.

### 14. Modifying GeneralInfoTabContainer Caused Page Freeze
**Problem:** Changing the `GeneralInfoTabContainer` from `"operation": "remove"` to `"operation": "merge"` with `"visible": true` and inserting a `crt.FlexContainer` inside it caused the page to spin infinitely.
**Why:** The original schema had removed the tab container. Re-adding it while other parts of the schema still referenced removed child elements likely caused a rendering loop in the Freedom UI framework.
**Fix:** Don't modify containers that were removed in the base schema. Work within the existing page structure or create new top-level containers.

### 15. Zone.js Conflict — Browser Freeze When Setting Data
**Problem:** After feeding data to the `<forecast-hierarchy>` custom element inside Creatio, the browser tab froze completely and became unresponsive.
**Why:** Our Angular 21 component bootstrapped with `createApplication()` uses Zone.js by default. When nested inside Creatio's own Angular app (which also uses Zone.js), setting properties on the custom element triggers Zone.js change detection in both apps simultaneously, creating an infinite feedback loop.
**Fix:** Use `provideZonelessChangeDetection()` in the Creatio entry point:
```ts
createApplication({
  providers: [provideZonelessChangeDetection()],
}).then((appRef) => {
  const el = createCustomElement(HierarchicalListComponent, { injector: appRef.injector });
  customElements.define('forecast-hierarchy', el);
});
```
This is **critical** for any Angular custom element that will run inside another Angular application (like Creatio). The component uses Angular signals for reactivity instead of Zone.js, so zoneless mode works perfectly.

### 16. Module Federation Wrapper Freezes the Page
**Problem:** Even with `provideZonelessChangeDetection()` on the AMD component, the `<usr-forecast-hierarchy>` wrapper component (from the Module Federation Angular 17 project) freezes the browser tab when placed on a page via the Freedom UI Designer.
**Why:** The Module Federation remote module bootstraps its own Angular 17 app inside Creatio's Angular shell via `bootstrapCrtModule()`. This creates a second Angular application context. Even though the inner `<forecast-hierarchy>` custom element uses zoneless change detection, the wrapper itself runs inside Creatio's Zone.js context and triggers infinite change detection cycles.
**Status:** Unresolved. The wrapper appears in the Freedom UI Designer (drag-and-drop works), but rendering it on an actual page freezes the browser.
**Workaround:** Use the AMD-loaded `<forecast-hierarchy>` custom element directly via DOM manipulation in a handler, without the Module Federation wrapper. This approach works — the button test confirmed the element loads, responds to data, and doesn't freeze.
**Possible fixes to investigate:**
- Run the wrapper component outside Angular's zone
- Use `ChangeDetectionStrategy.OnPush` on the wrapper
- Defer the inner `<forecast-hierarchy>` creation until after the wrapper stabilizes
- Skip the wrapper entirely and use a pure DOM-based approach for placing the component

### 17. Content Hash Not Updated After Rebuild
**Problem:** After rebuilding the JS file, Creatio served the cached old version because the hash in `_FileContentDescriptors.js` didn't match the new file.
**Fix:** Update the hash in `_FileContentDescriptors.js` after each rebuild, or use `clio push-pkg` which handles this automatically. The `build-creatio.js` script should ideally update this hash.

## File Content System — How It Works

Creatio serves static file content through a specific pipeline:

1. **`Pkg/{PackageName}/Files/`** — source files in the package
2. **`conf/content/_FileContentBootstraps.js`** — maps package names to bootstrap files
3. **`conf/content/_FileContentDescriptors.js`** — maps file paths to content hashes (for cache busting)
4. **`conf/content/{PackageName}/...`** — actual files served to the browser

When `push-pkg` installs a package, it:
- Copies files from the package archive to the database
- Updates `_FileContentBootstraps.js` and `_FileContentDescriptors.js`
- Copies files to `conf/content/` directory

When FSM is on and `pkg-to-file-system` is run:
- Database content is synced to `Pkg/` filesystem directory
- `conf/content/` is regenerated

**URL pattern:** `http://localhost:9090/conf/content/{PackageName}/path/to/file.js`
(Note: no `/0/` prefix — that's for API endpoints only)

## Deployment Flow — What Actually Works

### Initial Setup
1. Deploy Creatio locally via `clio deploy-creatio`
2. Enable FSM: `clio turn-fsm on`
3. Install cliogate: `clio install-gate`
4. Create or push the component package: `clio push-pkg`
5. Unlock the package: `clio unlock-package`
6. Generate the Module Federation wrapper project: `clio new-ui-project`

### Development Cycle — Core Component
1. Edit component source code in `src/`
2. Run `npm run build:creatio` — builds AMD-wrapped JS
3. Copy to deployed Creatio (build script does this automatically)
4. `clio push-pkg` to update file content descriptors and hashes
5. Hard-refresh (Cmd+Shift+R) the Creatio page

### Development Cycle — Designer Wrapper
1. Edit wrapper component in `projects/forecast_hierarchy/`
2. Run `npm run build` in that directory
3. Copy build output to deployed Creatio locations (Pkg + conf/content)
4. `clio push-pkg` to register new files
5. Restart Creatio: `clio restart`

### Page Schema Changes
1. Edit schema JS files directly in `Pkg/{Package}/Schemas/{Schema}/`
2. Refresh the Creatio page — FSM picks up changes automatically
3. For changes made via the Creatio UI designer: `clio pkg-to-file-system` to sync back

### Full Rebuild + Deploy (both layers)
```bash
# 1. Build AMD core component
npm run build:creatio

# 2. Build Module Federation wrapper
cd projects/forecast_hierarchy && npm run build && cd ../..

# 3. Copy wrapper output to deployed Creatio
SRC="packages/UsrComponentPackage/Files/src/js/forecast_hierarchy"
DEST="creatio-pulled/creatio-local/Terrasoft.Configuration/Pkg/UsrComponentPackage/Files/src/js/forecast_hierarchy"
CONTENT="creatio-pulled/creatio-local/conf/content/UsrComponentPackage/src/js/forecast_hierarchy"
cp -r "$SRC" "$DEST" && cp -r "$SRC" "$CONTENT"

# 4. Push package (updates hashes and bootstraps)
clio push-pkg packages/UsrComponentPackage -e creatio-local
```
